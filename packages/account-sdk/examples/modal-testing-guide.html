Base Account SDK - Modal Testing Guide

This guide explains how to test the various modal/dialog components in the Base Account SDK. The SDK uses internal modals for important user interactions that require attention or decisions.

Modal Architecture

The SDK's modal system consists of:

  • Dialog Component: A Preact-based UI component
  • CSS Classes: Prefixed with -base-acc-sdk- to avoid conflicts
  • Z-index: Set to maximum (2147483647) to ensure visibility
  • Mobile Support: Bottom sheet on mobile portrait mode with drag-to-dismiss

Modal Types

1. Popup Blocked Modal

When it appears: When the browser blocks a popup window that the SDK needs to open for wallet interactions.

Title format: "{AppName} wants to continue in Base Account"

Message: "This action requires your permission to open a new window."

Actions:

  • Try again (Primary button)
  • Cancel (Secondary button)
How to trigger:
  1. Block popups in your browser settings
  2. Create an SDK instance
  3. Attempt to connect wallet or perform any action requiring a popup

2. Insufficient Balance Modal

When it appears: When a sub-account's spend permission balance cannot cover a transaction.

Title: "Insufficient spend permission"

Message: "Your spend permission's remaining balance cannot cover this transaction. Please choose how to proceed:"

Actions:

  • Edit spend permission (Primary button)
  • Use primary account (Secondary button)
How to trigger:
  1. Set up a sub-account with limited spend permission
  2. Attempt a transaction exceeding the permission limit

3. Add Owner Modal

When it appears: When the app has lost access to a user's account and needs re-authorization.

Title format: "Re-authorize {AppName}"

Message format: "{AppName} has lost access to your account. Please sign at the next step to re-authorize {AppName}"

Actions:

  • Continue (Primary button)
  • Not now (Secondary button)
How to trigger:
  1. Remove app's access to the account externally
  2. Attempt an operation requiring account access

Testing Modals Programmatically

üí° Tip: Use MutationObserver to detect when modals appear in the DOM. 
// Monitor for modal appearances
const observer = new MutationObserver((mutations) => {
    mutations.forEach((mutation) => {
        mutation.addedNodes.forEach((node) => {
            if (node.nodeType === 1 && node.classList?.contains('-base-acc-sdk-css-reset')) {
                const dialog = node.querySelector('.-base-acc-sdk-dialog-instance');
                if (dialog) {
                    console.log('Modal detected!');
                    const title = dialog.querySelector('.-base-acc-sdk-dialog-instance-content-title')?.textContent;
                    const message = dialog.querySelector('.-base-acc-sdk-dialog-instance-content-message')?.textContent;
                    console.log('Title:', title);
                    console.log('Message:', message);
                }
            }
        });
    });
});

observer.observe(document.body, {
    childList: true,
    subtree: true
});

CSS Selectors for Testing

Use these selectors to find modal elements:

  • .-base-acc-sdk-css-reset - Root container
  • .-base-acc-sdk-dialog-container - Dialog container
  • .-base-acc-sdk-dialog-instance - Individual dialog
  • .-base-acc-sdk-dialog-instance-content-title - Dialog title
  • .-base-acc-sdk-dialog-instance-content-message - Dialog message
  • .-base-acc-sdk-dialog-instance-button - Action buttons

Mobile Testing

On mobile devices in portrait mode:

  • Modals appear as bottom sheets
  • Support drag-to-dismiss gesture
  • Drag down more than 100px to dismiss
  • Animated transitions for smooth UX
